3. Démarche de Conception de l'Architecture

Méthodologie: Modèle C4 + TOGAF

Notre démarche de conception est cohérente avec les activités d'architecture logicielle et suit une approche progressive du stratégique vers le détaillé.

Activités d'Architecture Logicielle

Phase 1: Analyse et Compréhension (Jours 1-2)

  • Étude du contexte: Analyse du système existant Good Food
  • Identification des besoins métier: Entretiens avec les parties prenantes
  • Analyse des exigences: Extraction des exigences fonctionnelles et non-fonctionnelles
  • Identification des contraintes: Techniques, budgétaires, temporelles

Livrables: Document d'analyse existant, matrice exigences

Phase 2: Architecture Métier - Niveau 1 Contexte (Jour 2)

  • Cartographie de l'écosystème: Identification de tous les acteurs et systèmes
  • Définition des frontières: Ce qui est dans/hors périmètre
  • Vue d'ensemble stratégique: Compréhensible par la Direction
  • Parcours utilisateur: Workflows Client, Franchisé, Livreur

Livrables: Diagrammes C4 Niveau 1 (Contexte)

Phase 3: Architecture Applicative - Niveau 2 Conteneurs (Jour 3)

  • Décomposition en services: Identification des conteneurs (applications, bases de données)
  • Définition des technologies: Choix des stacks techniques
  • Architecture applicative: Organisation des services métier
  • Flux de données: Communication entre conteneurs

Livrables: Diagrammes C4 Niveau 2 (Conteneurs)

Phase 4: Architecture Technique - Niveau 3 Composants (Jours 3-4)

  • Détail des composants internes: Décomposition de chaque service
  • Patterns de conception: Application des patterns architecturaux
  • Architecture hexagonale: Séparation métier/infrastructure
  • Flux techniques détaillés: Séquences d'appels

Livrables: Diagrammes C4 Niveau 3 (Composants), Architecture hexagonale

Phase 5: Comparaison et Justification (Jour 4)

  • Évaluation des styles architecturaux: Matrice de comparaison
  • Comparaison des technologies: Matrices de décision
  • Analyse coûts/bénéfices: TCO et ROI
  • Justification des choix: Documentation des décisions

Livrables: Matrices de comparaison, ADR (Architecture Decision Records)

Phase 6: Validation et Documentation (Jour 5)

  • Vérification de cohérence: Alignement avec exigences
  • Revue par pairs: Validation technique
  • Documentation finale: Guide complet d'architecture
  • Présentation: Préparation support pour décideurs

Livrables: Document final, présentation, diagrammes exportés

Frameworks et Outils Utilisés

C4 Model

  • Contexte (stratégique)
  • Conteneurs (fonctionnel)
  • Composants (détaillé)

Pattern de conception

  • Séparation métier/infra
  • Testabilité maximale
  • Évolutivité garantie
  • Indépendance technologique

Cohérence avec les Activités d'Architecture

Notre démarche suit rigoureusement les étapes standards de conception d'architecture logicielle:

  1. Analyse: Compréhension du contexte et des besoins
  2. Conception: Du général (contexte) vers le spécifique (composants)
  3. Évaluation: Comparaison des alternatives et justification des choix
  4. Validation: Vérification de l'alignement avec les exigences
  5. Documentation: Traçabilité des décisions

Notre choix combine les 3 meilleurs styles pour maximiser les bénéfices:

Justification Détaillée par Rapport au Contexte

1. Pourquoi Microservices?

✅ Scalabilité (5/5)

Besoin: Passer de 150 à 500+ restaurants

Solution: Chaque service scale indépendamment selon la charge

Exemple: Service Commande peut avoir 10 instances pendant le pic déjeuner, Service Analytics seulement 2

✅ Maintenabilité (5/5)

Problème actuel: Code legacy monolithique impossible à maintenir

Solution: Services indépendants, équipes autonomes

Bénéfice: Refonte progressive, pas de big-bang

✅ Évolutivité (5/5)

Besoin: Ajouter rapidement de nouvelles fonctionnalités

Solution: Nouveaux services sans impact sur l'existant

Exemple: Ajout service Réservations sans toucher aux Commandes

✅ Résilience

Problème actuel: Panne globale si un composant tombe

Solution: Isolation des pannes par service

Exemple: Panne du service Analytics n'impacte pas les commandes

2. Pourquoi Architecture Hexagonale?

✅ Testabilité (5/5)

Problème: Code legacy non testable

Solution: Cœur métier isolé, tests unitaires purs

Objectif: 80% de couverture de tests

✅ Indépendance Techno

Besoin: Pouvoir changer de base de données, APIs externes

Solution: Infrastructure interchangeable via ports/adapters

Exemple: Passer de SQL Server à PostgreSQL sans toucher au métier

✅ Clarté du Code

Problème: Code spaghetti actuel

Solution: Séparation claire métier/infrastructure

Bénéfice: Onboarding développeurs facilité

✅ Évolution Métier

Besoin: Adapter rapidement aux changements métier

Solution: Règles métier centralisées et isolées

Exemple: Changement règles de promotion sans impact technique

3. Pourquoi Event-Driven?

✅ Découplage (5/5)

Problème: Couplage fort actuel entre composants

Solution: Communication asynchrone via événements

Exemple: Service Commande publie "CommandeValidée", plusieurs services réagissent

✅ Performance (5/5)

Problème actuel: Temps de réponse > 5s

Solution: Traitement asynchrone des tâches longues

Exemple: Envoi email/SMS en background, réponse immédiate à l'utilisateur

✅ Traçabilité

Besoin: Audit complet des opérations

Solution: Event Store conserve l'historique

Exemple: Replay des événements pour debug ou analytics

✅ Synchronisation ERP

Problème actuel: Sync 1x/jour avec erreurs

Solution: Événements publiés en temps réel vers ERP

Objectif: Synchronisation < 5 minutes

4. Pourquoi PAS Monolithique?

❌ Inadapté aux Exigences de Good Food

  • Scalabilité insuffisante (2/5): Ne peut pas scaler indépendamment par fonction → Impossible de supporter 500+ restaurants
  • Maintenabilité faible (2/5): Code legacy actuel déjà ingérable → Reproduire le même problème
  • Évolutivité limitée (2/5): Ajout de fonctionnalités impacte tout le système → Déploiements risqués
  • Rigidité technique: Impossible de moderniser progressivement → Refonte big-bang trop risquée
  • Performance sous charge: Malgré le 4/5, tout scale ensemble → Gaspillage de ressources

✅ Cohérence avec le Contexte Good Food

Les styles retenus répondent spécifiquement aux problématiques identifiées:

  • Croissance: 150 → 500+ restaurants → Microservices scalable
  • Code legacy: Impossible à maintenir → Architecture hexagonale testable
  • Sync ERP: 1x/jour insuffisant → Event-driven temps réel
  • Performance: > 5s actuellement → Cache, async, optimisations
  • Mobilité: Pas d'app mobile → Architecture API-first
  • Sécurité: Failles multiples → Architecture sécurisée by design
